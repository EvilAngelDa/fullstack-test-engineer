# 接口用例 vs 前端功能用例（强制分层）

## 一句话

- **接口**：按**需求/模块描述**把数据返回对。  
- **前端**：按**设计稿**画对、按**需求**交互对；**只根据接口有没有数据、数据正不正常**决定展示与交互。  
- **登录**：一般只测接口鉴权；前端不写「匿名能不能看」——有数据就展示。

## 对照

| 问题 | 归哪边 |
|------|--------|
| 电车必须返回哪几个 metricName？ | **接口**（需求清单） |
| 页面卡片要不要横滑、吸顶、2 行展开？ | **前端**（设计+交互） |
| 返回 3 条指标页面展几条？ | **前端**：几条展几条 |
| 完整数据时接口少了需求项？ | **接口缺陷** |
| 有数据/空数据/失败显隐？ | **前端**（按响应形态；场景要拆） |
| 是否允许未登录调通？ | **接口** |

## 前端四段式（推荐用例分组）

1. **正常数据 — 展示**（对齐 UI + 字段与接口一致）  
2. **正常数据 — 交互**（Tab、展开、切换…）  
3. **空数据 — 展示/交互**（不展模块/Tab…）  
4. **异常数据/失败 — 展示/交互**（兼容、不白屏、可浏览其它模块…）

## 禁止

- 接口预期里出现：吸顶、省略号、卡片样式、窄屏适配  
- 前端预期里出现：必须登录 / 匿名可浏览（在数据驱动显隐前提下）  
- 用一条「异常不展示」合并风控、网络、500、空数据、串号  

## 字段精细化 vs PRD（已锁定）

- 模块 PRD 写明了 → 听 PRD，备注「PRD 覆盖默认矩阵」  
- PRD 没写 → 用 `field-refined-case-rules.md` / display-field 默认  
- 不因模块特例删除 skill 全局默认规则  

## 数据驱动显隐 vs 用户态交互（补充，不覆盖上表）

| 对象 | 是否跟登录/token | 说明 |
|------|------------------|------|
| **模块内容显隐**（卡片列表、正文区） | 一般 **否** | 仍数据驱动：有数据展示、无数据隐藏 |
| **用户个人态**（是否已点「有帮助/已采纳」） | **是（用户维度）** | 有 token 才查 user-state；无 token 默认未操作态 |
| **需登录的动作**（提交采纳、进会话） | **是** | 无 token → 跳登录 → 成功后**回流续原动作** |

1. **不冲突旧规则：**「前端不写匿名能不能看模块」仍然成立——**看内容**跟数据；**个人操作态/提交**可单独写登录链路。  
2. 无 token：**不请求**用户态 query；UI 固定默认未点击文案。  
3. 未登录点击 → 登录取消/失败：不伪造已操作成功态。  
4. 接口实现了取消/点踩但 PRD 未要求：**前端不写入口**；接口 cancel 不作前端验收主路径。  
5. 用户态维度模型常见：`userId × objectUid × sourceType`（字段名以当前契约为准）。
